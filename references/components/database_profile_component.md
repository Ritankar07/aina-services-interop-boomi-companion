## Purpose
Defines the SQL operation and the request/response data structure for the **Database (Legacy) connector**. A single Database Profile holds one statement: the SQL (or stored-procedure call), the bind parameters/conditions that feed it, and — for reads — the result-set columns that become the response document.

This is **not** the same as Database V2. The Database V2 connector uses JSON request/response profiles (see `databasev2_connector_operation_component.md`). A `profile.db` component is consumed only by the Database (Legacy) connector.

## Component Type
- **Type**: `profile.db`
- **Root element**: `<DatabaseProfile>`

```xml
<DatabaseProfile strict="true" version="2">
```
- `strict` — boolean (default `false`); typically `true`
- `version` — profile model version (int, default `1`); current profiles use `2`

## Connector Context
A Database (Legacy) operation references the profile by ID. A GET (read) operation:
```xml
<Operation>
  <Configuration>
    <DatabaseGetAction batchCount="0" maxRows="0">
      <ReadProfile profileId="{profile-db-guid}"/>
    </DatabaseGetAction>
  </Configuration>
</Operation>
```
Inside a process, the connector action references the profile through the `parameter-profile` attribute:
```
parameter-profile="EMBEDDED|databaseparameterchooser|{profileId}|{operationId}"
```

## XML Structure

```xml
<?xml version="1.0" encoding="UTF-8"?>
<bns:Component xmlns:bns="http://api.platform.boomi.com/"
               xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
               componentId=""
               name="Profile_Name"
               type="profile.db"
               folderId="{FOLDER_GUID}">
  <bns:encryptedValues/>
  <bns:object>
    <DatabaseProfile strict="true" version="2">
      <ProfileProperties>
        <DatabaseGeneralInfo executionType="dbread"/>
      </ProfileProperties>
      <DataElements>
        <DBStatement isNode="true" key="2" name="Statement"
                     statementType="select" storedProcedure="" tableName="">
          <!-- DBFields / DBParameters / DBConditions as required by statementType -->
          <sql>SELECT ...</sql>
        </DBStatement>
      </DataElements>
    </DatabaseProfile>
  </bns:object>
</bns:Component>
```

## Profile Properties
`<DatabaseGeneralInfo>` carries `executionType`:

| executionType | Used by statement types |
|---------------|-------------------------|
| `dbread`  | `select`, `spread` |
| `dbwrite` | `standardinsertupdatedelete`, `dynamicinsert`, `dynamicupdate`, `dynamicdelete`, `spwrite` |

An optional `description` attribute is also valid on `DatabaseGeneralInfo`.

## DBStatement
One `<DBStatement>` per profile (always `key="2"`, `name="Statement"`). Child element order: `DBFields`, `DBParameters`, `DBConditions`, `sql`.

| Attribute | Notes |
|-----------|-------|
| `statementType` | One of `select`, `spread`, `standardinsertupdatedelete`, `dynamicinsert`, `dynamicupdate`, `dynamicdelete`, `spwrite` |
| `tableName` | Target table. **Required** for `dynamicinsert`, `dynamicupdate`, and `dynamicdelete` (these generate SQL from it); empty for the other modes |
| `storedProcedure` | The procedure call string for `spread` / `spwrite`; empty otherwise |

### What executes
The `<sql>` element is always present, but it is **not** the executed statement for every mode:

| Mode | What runs at request time |
|------|---------------------------|
| `select` | The `<sql>` text (the read query) |
| `standardinsertupdatedelete` | The literal `<sql>` text, with `DBFields` bound positionally to its `?` placeholders |
| `dynamicinsert`, `dynamicupdate`, `dynamicdelete` | SQL **generated by the connector** from `tableName` + `DBFields` (columns/SET) + `DBConditions` (WHERE). The stored `<sql>` is **ignored** — it may be empty (`<sql/>`) or hold a preview string, but it does not execute |
| `spread`, `spwrite` | The `storedProcedure` (the bare proc name); the connector builds the callable and `<sql/>` is empty (see spread / spwrite) |

For `standardinsertupdatedelete` only the **first** statement in `<sql>` runs — multi-statement (`;`-separated) batches are not supported.

## Read Statements

### select
Standard SQL read. `DBFields type="result_set"` lists the **output columns** (the response document); `DBParameters` lists the `?` input binds (empty when the SQL has none).

result_set elements capture the query's result columns by **ordinal position** — the Nth element receives the Nth selected column. Element names are just labels (for downstream maps); they need not match the SQL column names. List them in SELECT order. Listing fewer elements than selected columns captures only the leading columns; the rest are discarded.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="select" storedProcedure="" tableName="">
  <DBFields isNode="true" key="3" name="Fields" type="result_set">
    <DatabaseElement dataType="number" enforceUnique="false" isMappable="true" isNode="true" key="5" mandatory="false" name="BusinessEntityID">
      <DataFormat><ProfileNumberFormat/></DataFormat>
    </DatabaseElement>
    <DatabaseElement enforceUnique="false" isMappable="true" isNode="true" key="6" mandatory="false" name="FirstName">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <DBParameters isNode="true" key="4" name="Parameters"/>
  <sql>SELECT BusinessEntityID, FirstName, LastName, EmailPromotion
FROM Person.Person
WHERE PersonType = 'EM'</sql>
</DBStatement>
```

An empty `<DBFields type="result_set"/>` is accepted and the read still runs (emitting one row per result row), but the rows come back **empty** — the connector returns no column data and does not fall back to the selected columns. Enumerate the result_set columns to get usable data.

**Authoring the columns.** The `result_set` columns (and the SQL) are authored directly in this XML. The Boomi GUI's **Import Wizard** can auto-generate them by browsing the database schema through a deployed runtime, but that import is a **build-time GUI convenience** — there is no platform API for it. A profile built via the API hand-authors the `result_set` columns and reads fully at request time; the import is not a request-time prerequisite. (The wizard also does not support `standardinsertupdatedelete`, and does not populate cursor/result-set columns for stored procedures — those are always hand-authored.)

### spread (stored-procedure read)
`storedProcedure` is **what executes** and must be the **bare, schema-qualified procedure name** — no `EXEC`, no `?` placeholders. The connector builds the JDBC callable `{? = call proc(?…)}` itself, reserving parameter 1 for the procedure's return value and binding the `DBParameter`s (`mode="in"`, `@`-prefixed names) positionally after it. `<sql/>` is **empty**. `DBParameters` holds the input arguments; `DBFields type="result_set"` holds the returned columns (optionally including `@RETURN_VALUE`).

Adding `EXEC` or a trailing `?` to `storedProcedure` is the classic mistake — it adds an extra placeholder and fails at request time with *"The value is not set for the parameter number N+1"*. Input values are bound on the **process step** (`valueType="current"`), not here — see `database_connector_step.md`.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="spread"
             storedProcedure="dbo.uspGetEmployeeManagers" tableName="">
  <DBFields isNode="true" key="3" name="Fields" type="result_set">
    <DatabaseElement dataType="number" enforceUnique="false" isMappable="true" isNode="true" key="5" mandatory="false" name="RecursionLevel">
      <DataFormat><ProfileNumberFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <DBParameters isNode="true" key="4" name="Parameters">
    <DBParameter dataType="number" isMappable="false" isNode="true" key="9" mode="in" name="@BusinessEntityID">
      <DataFormat><ProfileNumberFormat/></DataFormat>
    </DBParameter>
  </DBParameters>
  <sql/>
</DBStatement>
```

## Write Statements

### standardinsertupdatedelete (standard)
You supply the full INSERT/UPDATE/DELETE SQL with `?` placeholders, and this literal `<sql>` is what executes. `DBFields` define the parameters: **fields and `?` placeholders must match in sequence and number.** `tableName` is empty. Only the **first** statement runs — `;`-separated batches are not supported.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="standardinsertupdatedelete" storedProcedure="" tableName="">
  <DBFields isNode="true" key="3" name="Fields">
    <DatabaseElement enforceUnique="false" isMappable="true" isNode="true" key="4" mandatory="false" name="CustomerNum">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DatabaseElement>
    <DatabaseElement enforceUnique="false" isMappable="true" isNode="true" key="5" mandatory="false" name="CustomerName">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <sql>INSERT INTO Customer (CustomerNum, CustomerName)
VALUES(?, ?)</sql>
</DBStatement>
```

### dynamicinsert
`tableName` (required) names the target table; `DBFields` list the columns to insert. The connector generates the `INSERT` from these — the `<sql>` shown below is not executed.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="dynamicinsert"
             storedProcedure="" tableName="HumanResources.Department">
  <DBFields isNode="true" key="3" name="Fields">
    <DatabaseElement enforceUnique="false" isMappable="true" isNode="true" key="5" mandatory="false" name="Name">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DatabaseElement>
    <DatabaseElement dataType="datetime" enforceUnique="false" isMappable="true" isNode="true" key="7" mandatory="false" name="ModifiedDate">
      <DataFormat><ProfileDateFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <sql>INSERT INTO HumanResources.Department (Name, GroupName, ModifiedDate)
VALUES (?, ?, ?)</sql>
</DBStatement>
```

### dynamicupdate
`tableName` (**required**) names the target table; `DBFields` hold the columns to SET; `DBConditions` hold the WHERE clause. The connector generates `UPDATE <tableName> SET <DBFields> WHERE <DBConditions>`; the `<sql>` is not executed.

A blank `tableName` is an **invalid configuration** — it fails at request time with *"Table name must not be blank for a dynamic update"*, and a populated `<sql>` does not substitute for it. A profile that pairs an empty `tableName` with a populated `<sql>` does not run.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="dynamicupdate" storedProcedure="" tableName="HumanResources.Employee">
  <DBFields isNode="true" key="3" name="Fields">
    <DatabaseElement enforceUnique="false" isMappable="true" isNode="true" key="4" mandatory="false" name="JobTitle">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <DBConditions isNode="true" key="5" name="Conditions">
    <DBCondition conditionOperator="equal" dataType="number" isMappable="true" isNode="true" key="6" name="BusinessEntityID">
      <DataFormat><ProfileNumberFormat/></DataFormat>
    </DBCondition>
  </DBConditions>
  <sql/>
</DBStatement>
```

### dynamicdelete
`tableName` (required) names the target table; `DBConditions` hold the WHERE clause. No `DBFields`. The connector generates the `DELETE` from these; `<sql>` is empty and not executed.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="dynamicdelete"
             storedProcedure="" tableName="HumanResources.Department">
  <DBConditions isNode="true" key="4" name="Conditions">
    <DBCondition conditionOperator="equal" isMappable="true" isNode="true" key="5" name="DepartmentID">
      <DataFormat><ProfileCharacterFormat/></DataFormat>
    </DBCondition>
  </DBConditions>
  <sql/>
</DBStatement>
```

### spwrite (stored-procedure write)
`storedProcedure` is **what executes** and must be the **bare, schema-qualified procedure name** (no `EXEC`, no `?` — same form as spread); the connector builds the callable itself and `<sql/>` is empty. `DBFields` list the procedure's input arguments. See `database_connector_step.md` for input binding.

```xml
<DBStatement isNode="true" key="2" name="Statement" statementType="spwrite"
             storedProcedure="HumanResources.uspUpdateEmployeeHireInfo" tableName="">
  <DBFields isNode="true" key="3" name="Fields">
    <DatabaseElement dataType="number" enforceUnique="false" isMappable="true" isNode="true" key="4" mandatory="false" name="BusinessEntityID">
      <DataFormat><ProfileNumberFormat/></DataFormat>
    </DatabaseElement>
    <DatabaseElement dataType="datetime" enforceUnique="false" isMappable="true" isNode="true" key="5" mandatory="false" name="RateChangeDate">
      <DataFormat><ProfileDateFormat/></DataFormat>
    </DatabaseElement>
  </DBFields>
  <sql/>
</DBStatement>
```

## DBFields, DBParameters, DBConditions
Three container elements hold the typed nodes for a statement. Each container has `key`, `name`, `isNode="true"`.

| Container | Holds | Used by |
|-----------|-------|---------|
| `DBFields` | `DatabaseElement` — result-set output columns (reads) or input columns/params (writes) | all except `dynamicdelete` |
| `DBParameters` | `DBParameter` — `?` input binds | `select`, `spread` |
| `DBConditions` | `DBCondition` — WHERE-clause terms | `dynamicupdate`, `dynamicdelete` |

`DBFields` carries `type="result_set"` on read profiles. (The other valid value is `parameter`.)

## DatabaseElement Attributes
`DatabaseElement` (in `DBFields`), `DBParameter` (in `DBParameters`), and `DBCondition` (in `DBConditions`) share a common base:

| Attribute | Notes |
|-----------|-------|
| `key` | Unique int |
| `name` | Column / parameter name |
| `isNode` | `true` |
| `isMappable` | `true` for mapped fields; `false` seen on stored-proc input params |
| `dataType` | `character` (default), `number`, `datetime`, `clob`, `blob`. May be omitted when authoring character elements; profiles built through the Boomi UI write `dataType="character"` explicitly |
| `mandatory` | `DatabaseElement` only; boolean, default `false` |
| `enforceUnique` | `DatabaseElement` only; boolean |
| `conditionOperator` | `DBCondition` only; `equal` (default), `notequal`, `greaterthan`, `greaterthanequal`, `lessthan`, `lessthanequal` |
| `mode` | `DBParameter` only; `in` (default), `out`, `in_out`, `return`, `unknown` |
| `type` | `DBParameter` only; `value` (default), `cursor` |

## Data Formats
Each node carries a `<DataFormat>` matching its `dataType`. The formats are typically empty (no attributes); the optional attributes below are valid.

| dataType | DataFormat element | Optional attributes |
|----------|--------------------|---------------------|
| character | `<ProfileCharacterFormat/>` | — |
| number | `<ProfileNumberFormat/>` | `numberFormat` (default `#`), `impliedDecimal` (default `0`), `signedField` (default `false`) |
| datetime | `<ProfileDateFormat/>` | `dateFormat` |
| (boolean) | `<ProfileBooleanFormat/>` | — |

At request time, values fed into a `datetime` field must use the Boomi internal date/time format `yyyyMMdd HHmmss.SSS` (e.g. `20260602 120000.000`); other formats fail with an *"Unparseable date"* error.

## Key Generation
- Keys are unique ints across the profile, assigned in creation order.
- `DBStatement` is `key="2"`; the first container is typically `key="3"`.
- Keys may have gaps after fields are removed (e.g. `4, 5, 6, 12`). Gaps are harmless — only uniqueness matters.
